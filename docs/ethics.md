# 伦理原则与使用边界

## 为什么坚持 human-in-the-loop

课程评价不是单纯的表单录入任务。它可能影响教师获得的反馈、教学改进方向和学校内部决策，也承载着评价者本人的真实判断。因此，本项目采用 **human-in-the-loop（人在回路中）** 的设计：脚本可以减少重复点击、提供文字起点，但不能代替用户观察课程、形成判断、检查内容和作出最终提交决定。

这条原则体现在具体行为上：

1. 脚本运行前会说明即将进行的预填操作，并要求用户确认。
2. 脚本只处理可见、可编辑的 Element UI 下拉框和开放式文本框。
3. 已经有文字的评价框默认不会被覆盖，以保护用户已写内容。
4. 完成后会展示发现、成功和跳过的数量，提醒用户逐项复核。
5. 最终保存或提交始终由用户手动完成。

辅助填写不能证明页面中的每一项都与脚本预期相同。网页结构、题目顺序和评分措辞都可能变化，人工复核是发现错误和纠正不合适表述的最后一道必要防线。

## 为什么不自动提交

自动提交会把“提高录入效率”变成“代替用户作决定”，也会产生多类风险：

- 用户可能尚未检查脚本选择的分值是否符合真实感受。
- 模板或 AI 草稿可能不适合某门课程、某位教师或某个具体问题。
- 页面变化可能导致字段匹配错误，而自动提交会让错误直接生效。
- 连续或批量提交会削弱课程评价的真实性，并可能违反学校制度。
- 提交通常是不可轻易撤销的外部状态变更，不应由辅助脚本擅自完成。

因此，代码不查找、不触发学校页面上的“保存”“提交”“确认”等操作，并在通用点击工具中对这些文字设置额外拦截。本项目也不提供批量提交、自动翻页提交或绕过确认的能力。

## 为什么 AI 文本必须由用户确认

AI 生成内容具有不确定性。即使提示词要求真实、礼貌和建设性，模型仍可能写出过度推断、空泛套话或用户并不认同的内容。AI 也没有亲自上过这门课程，不能拥有用户的真实体验，更不应替用户作最终评价。

AI 版遵循以下流程：

1. AI 模式默认关闭，用户必须主动开启。
2. 用户点击“生成草稿”并确认隐私提示后，脚本才会访问其配置的 API。
3. 生成结果只显示在脚本面板的草稿框中，不会自动写入学校页面。
4. 用户可以修改、删除或放弃草稿。
5. 只有用户主动点击“预填页面”，草稿才会写入仍为空白的文本框。
6. 写入后仍须用户再次检查，并手动保存或提交。

AI 的合适角色是写作辅助，而不是评价主体。

## 隐私原则

- Lite 版不调用任何外部接口。
- AI 版没有项目方的数据收集后端。
- 只有在用户主动生成草稿时，课程名、教师名、页面评价问题和用户补充才会发送到用户配置的 API 服务。
- API Key 保存在 Tampermonkey 的脚本存储中，不写入源代码；但它仍属于敏感凭据，用户应妥善管理并了解所选 API 服务的隐私政策。
- 不应在用户补充中填写学号、手机号、身份证号、登录凭证、Cookie、Token 或其他无关敏感信息。

## 可接受与不可接受的使用

可接受：减少重复选择、生成可编辑草稿、在提交前认真复核并按真实体验修改。

不可接受：代替他人评价、伪造体验、批量自动提交、绕过登录或验证码、尝试突破权限、利用脚本收集或外传学校页面数据。
